# LyricDock 开发计划

## 目标

做一个在 `niri / Wayland` 下可长期置顶显示的 Spotify 桌面歌词程序。

核心效果：

- 左侧固定显示歌词
- 始终位于普通窗口上层
- 跟随工作区切换持续存在
- 作为浮动覆盖层显示，不挤占其他程序空间
- 根据 Spotify 播放进度同步高亮歌词
- 歌词按需从网易云获取，并缓存到本地

## 当前已经验证通过的部分

我们已经确认：

- 可以从 `Spotify MPRIS` 读取当前歌曲信息
- 可以拿到歌曲名、歌手、专辑、总时长、播放进度、播放状态
- 可以通过本地调用 `NeteaseCloudMusicApi` 搜索并下载歌词
- 可以把歌词保存到仓库中的本地缓存目录
- 可以在 Slint 界面里按播放进度做主歌词高亮推进
- 可以在锁定/解锁态之间切换交互模式（锁定态穿透，解锁态可编辑）

这说明整条数据链路是可行的。

## 技术路线

### 1. 显示层

使用 `Wayland layer-shell` 做歌词窗口，而不是普通窗口。

原因：

- 普通窗口不适合实现常驻桌面歌词
- `layer-shell` 更接近状态栏、通知、桌面挂件这类组件
- 在 `niri` 中更容易实现“长期置顶、跨工作区可见”

预期行为：

- 默认作为浮动覆盖层显示
- `exclusive_zone = 0`
- 不为其他窗口保留布局空间
- 默认使用 `top` 或 `overlay` 层
- 如有需要，后续支持切换到 `overlay` 层
- 锚定左侧显示
- 不参与平铺布局

### 2. 播放信息层

通过 `MPRIS` 监听 Spotify：

- 当前歌曲元数据
- 播放 / 暂停状态
- 跳转进度
- 当前播放位置

策略：

- 曲目切换时重新查歌词
- 平时用轻量定时器更新高亮位置
- 不做高频网络请求

### 3. 歌词获取层

第一版优先接入网易云歌词。

处理流程：

1. 读取当前歌曲信息
2. 清洗标题和歌手文本
3. 搜索网易云候选歌曲
4. 对候选打分
5. 选择最佳匹配
6. 下载歌词
7. 解析 `.lrc`

后续可扩展：

- `LRCLIB`
- 本地自定义歌词
- 多来源 fallback

### 4. 缓存层

当前目录结构：

- `配置.toml` 指定歌词根目录
- 歌词根目录下分为：
  - `导入歌词/`
  - `缓存歌词/`

后续再做：

- 缓存数量上限
- 自动清理旧缓存
- 更完整的索引管理
- 更强的重名冲突处理

当前状态：

- 目前已经可以通过手填 `配置.toml` 指定歌词根目录
- 目前已经区分手动导入歌词与自动缓存歌词
- 目前已经有按总大小清理 `缓存歌词/` 的基础版本
- 自动缓存文件名已经改成更适合人工识别的 `歌手 - 歌曲名`

缓存内容至少包括：

- 原始歌曲信息
- 匹配结果
- 原始歌词文本
- 解析后的结构化歌词
- 写入时间
- 最后访问时间

### 5. UI 层

第一版 UI 目标：

- 左侧窄面板
- 显示当前行和前后几行
- 当前行高亮
- 暂停时冻结
- seek 时重新定位

当前进度（已完成）：

- 当前行 + 下一行双行布局
- 双行模式改成交错接力，不再按两句整组翻页
- 主行分层渲染（未播放层 + 已播放层）
- 按时间推进的动态高亮
- 已接入 `.yrc` 逐字歌词，支持在有逐字时间轴时做更细粒度的 karaoke 高亮
- 纯色/渐变高亮已支持配置切换，默认纯色
- 解锁态拖动/缩放、锁定态收敛交互
- 锁定态真正鼠标穿透
- 两行居中排版，长行按进度滚动
- 顶部工具条改成 open-orpheus 风格的图标按钮，支持锁定、字体大小调节

先不做：

- 复杂主题配置
- 逐字 karaoke 动画

更新说明：

- 点击穿透与拖拽/缩放已完成基础可用版本
- 当前鼠标穿透实现依赖仓库内附带的 `vendor/layer-shika-adapters`
- 后续如果要去掉 `vendor`，需要等待上游暴露更合适的底层 surface API，或自行维护 fork
- 后续重点转为：继续优化 karaoke 呈现细节，而不是再补底层数据链

## 第一阶段任务拆分

### 阶段 1：项目骨架

- 选定最终技术栈
- 建立程序目录结构
- 跑通最小窗口
- 在 `niri` 上显示一个左侧固定、浮动覆盖、不占空间的 layer-shell 窗口

### 阶段 2：播放监听

- 接入 Spotify `MPRIS`
- 获取歌曲信息
- 获取播放状态和进度
- 处理切歌与 seek

### 阶段 3：歌词链路

- 整理当前网易云抓词脚本
- 抽出可复用的匹配和下载模块
- 支持读取缓存
- 未命中时联网获取歌词

### 阶段 4：歌词展示

- 解析 LRC
- 根据进度计算当前行
- 在窗口中滚动显示歌词
- 完成基础同步效果

阶段 4 当前状态：

- 已完成：LRC 解析、当前行定位、主行进度高亮
- 进行中：更丝滑的时间插值与逐字进度
- 未开始：翻译行和音译行并行渲染

### 阶段 5：缓存完善

- 设计缓存元数据
- 增加索引文件
- 增加缓存清理策略
- 为后续更完整的设置管理做准备

## 我建议的实现顺序

为了尽快看到可运行结果，建议这样推进：

1. 先做 `layer-shell` 窗口
2. 再接 `MPRIS`
3. 再整合歌词获取和缓存
4. 最后做歌词滚动和样式

这样能最快得到一个“屏幕上真的有歌词”的版本。

## 当前决定

当前先不做：

- 缓存目录前端选择
- 设置页面
- 多歌词源切换
- 高级动画
- 更复杂的缓存策略

当前优先做：

- 能显示
- 能同步
- 能缓存
- 能在 `niri` 上常驻

新增优先级：

- 可读性优先于单一风格复刻
- 通过配置驱动样式（颜色、描边、透明度、效果模式）

## 样式目标（当前版本）

目标定位：

- 优先保证亮色与复杂背景下清晰可读
- 支持两种可切换效果：`flat`（平面全描边）和 `floating`（轻微浮动）
- 关键样式参数优先配置化，而不是硬编码

分阶段任务：

### 阶段 A：可读性基线

- 强化描边对比度与厚度
- 保证歌词默认不透明
- 在亮色背景下完成可读性验收

### 阶段 B：样式配置化

- 暴露 `lyric_effect`、`stroke_width`、`stroke_color`、`highlight_color` 等配置
- 保持配置项与 UI 实际生效一致
- 清理无效配置项，减少混淆

### 阶段 C：细节打磨

- 微调双行切换与逐字推进观感
- 在 `flat/floating` 两种模式下分别优化质感
- 增加不同背景场景的参数建议

## 下一步

下一步聚焦：

1. 完成 `flat` 模式亮色背景可读性参数预设
2. 补充配置项说明，确保“改配置即可调样式”
3. 针对 `floating` 模式再做轻量观感优化，不影响可读性基线
